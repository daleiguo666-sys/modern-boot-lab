## 1. 关键决定（已拍板，别再纠结）

| 维度 | 选择 | 备注 |
|---|---|---|
| 构建 | **Gradle Kotlin DSL** + Wrapper + toolchain | 对照 Maven 学 |
| 语言 | **Java 21**（temurin-21） | record / 虚拟线程 / pattern matching |

| 持久层 | **JPA**（已对比过 MyBatis-Plus，最终选定 JPA） | MP 代码已于 2026-06-15 移除 |

| API 文档 | springdoc-openapi（替代 springfox） | |
| JSON | Jackson（弃 fastjson，安全） | |
| 错误响应 | RFC 9457 ProblemDetail（替代 Results.fail） | |

- [x] **JPA 查询三层深入**（额外加餐，已写成可 curl 的真实接口）：
  - 派生查询 `findFirstByOrderByCreatedAtDesc` → `GET /api/jpa/widgets/latest`
  - `@Query`(JPQL) → `GET /api/jpa/widgets/in-stock?min=1`
  - Specification 动态查询 → `GET /api/jpa/widgets/search?status=&minQty=&sku=`
  - 接口投影 `WidgetSummary` → `GET /api/jpa/widgets/summaries`

- [x] **第 4 段 · JPA vs MyBatis-Plus 并排**：实体/Mapper/Service 三层对照、脏检查自动 UPDATE vs 显式 update、`LambdaQueryWrapper` 类型安全、选型建议
- [x] **第 5 段 · 工程化收尾**：配置绑定(record)、关 OSIV、ddl-auto vs Flyway、Actuator 可观测性、Testcontainers + `@Tag` 分层、Buildpacks 打包、单→多模块演进思路
- [x] **专题讨论**：为什么国内 MyBatis/MP 主流（路径依赖 + 高并发要 SQL 可控的真实理性，不只是历史）
- [x] **移除 MyBatis-Plus**（2026-06-15）：对比完两套 ORM 后选定 JPA（更声明式、脏检查自动 UPDATE、`@Version` 开箱即用）。删掉 `mp/` 整个包、`MyBatisPlusConfig`、`com.baomidou` 依赖与 `mybatis-plus` 配置。`/api/mp/...` 接口随之下线，主线只剩 JPA。
- [x] **第 6 段 · Flyway 版本化迁移**（待办 B 完成，2026-06-15）：`ddl-auto: update` → `validate`，schema 完全交给 Flyway。**已连真 Postgres 实跑验证全过**：空库建表(V1)→加索引(V2)、`flyway_schema_history` 历史表、增量幂等(重启显示 up-to-date 不重跑)、checksum 拦截(改 V1 启动直接失败)。依赖 `flyway-core` + `flyway-database-postgresql`，脚本在 `src/main/resources/db/migration/V1__init.sql`、`V2__add_status_index.sql`。
- [x] **第 7 段 · Testcontainers 集成测试**（待办 D 完成，2026-06-15）：`./gradlew integrationTest` 真起临时 `postgres:16-alpine` 容器跑测试、测完即焚（`docker ps -a` 无残留）。讲了 `@Container`/`@ServiceConnection`/`@Tag` 分层、用完即焚的隔离性。踩通 colima 两个真实坑：① socket 在非标准位置（`~/.testcontainers.properties` 配 `docker.host`）② ryuk 清理容器挂宿主 socket 在 vz/virtiofs 下不被支持（`build.gradle.kts` 里 `TESTCONTAINERS_RYUK_DISABLED=true` + socket override）。镜像走 DaoCloud 预拉。

- [x] **第 8 段 · 正规 CI/CD**（2026-06-16，计划外加餐，用户从「Jenkins 是否过时」问起）：用 GitHub Actions 搭真实流水线，全程实跑、推到 GitHub（账号 `daleiguo666-sys`，HTTPS + gh token），每步映射回公司自建 GitLab CI。
  - **M1 最小流水线** `.github/workflows/ci.yml`：`on` 触发、`jobs`/`steps`、`uses`(复现成动作) vs `run`(敲命令)、`checkout`+`setup-java`。踩坑：YAML 缩进（顶层键必须顶格）；读 CI 日志顺 `Caused by` 挖根因。
  - **可复现修正**：`application.yml` 原写死 `active: postgres` → CI 连不上库挂掉（典型「在我机器上好好的」）。改回默认 H2、postgres 交给环境变量。**确立原则：提交版配置必须自包含。**
  - **M2**：`cache: gradle` 依赖缓存（1m22s→42s）；`upload-artifact` 上传测试报告 + `if: always()`；actions 升 `checkout@v6`/`setup-java@v5`/`upload-artifact@v7`。
  - **M3 容器化**：多阶段 `Dockerfile`（JDK 编译→JRE 运行，850MB+→578MB）；本地 `docker build`/`run` 实跑；CI 加 `docker-image` job + `needs` 门禁。踩坑：`Dockerfile` 没 `git add`，CI 报 no such file（CI 只认 git 里的东西）。
  - **M4 推镜像到 GHCR**：`login-action`(GITHUB_TOKEN 临时令牌)+`metadata-action`(打 `sha`/`latest` 标签)+`build-push-action`；`permissions: packages:write` 最小权限；PR 只 build 不 push。产出不可变镜像 `ghcr.io/daleiguo666-sys/modern-boot-lab:sha-xxxx`。
  - **CD（仅概念，用户选「讲透即可」未动手）**：CI 终点=registry 里不可变镜像；CD 起点=把镜像跑到各环境。push 式(Jenkins/kubectl) vs GitOps(Argo CD/Flux 拉取 git 期望状态自动同步、selfHeal)。两仓库分离(应用仓 vs deploy 仓)、Kustomize overlays 按环境写死 tag。
  - 踩通国内两大网络坑（已持久化修复）：colima DNS 抽风 → `colima start --dns 223.5.5.5 --dns 8.8.8.8`；Docker Hub 拉取被掐 → `colima.yaml` 配 `registry-mirrors`（daocloud + 1ms）。
  - **专题（职业向）**：大公司 CI/CD 分工——平台/SRE 维护底座（runner、集群、Argo CD、CI 模板），程序员自己 own 服务的流水线/Dockerfile/部署清单，且「能读日志、出事自己修」是不可替代的核心技能，不是「大概知道就行」。

## 5. IDE 备注（已解决的坑，备查）

- 注解 `@Xxx` 变黑不高亮 = 开了 **Power Save Mode**（File 菜单关掉即恢复）。它省 CPU 不省内存。
- 省内存正确做法：禁用用不到的插件（PHP/Python/Mermaid）、一次只开一个项目、排除 `build/` 索引、别同开 2025.3 + 2026.1。
- 项目约定 **「禁用 Stream、一律 for 循环」只适用 werewolf-server**，本项目正常用 Stream（hook 已按路径收口）。
